# Адаптивная пространственная индексация

**Онлайн-выбор пространственного индекса для поиска соседей в фиксированном радиусе в динамических системах частиц.**

[English version](README.md)

`adaptive-spatial-indexing` — исследовательский проект на C++20 для сравнительного изучения пространственных индексов на динамических системах частиц.

Это **не демонстрация одной выбранной структуры данных**. Репозиторий задуман как экспериментальная среда для реализации, проверки и честного сравнения нескольких подходов к поиску соседей при разных распределениях частиц, моделях движения и аппаратных конфигурациях.

## Проблема

В динамической системе частицы постоянно меняют координаты. На каждом шаге необходимо находить все пары частиц, расстояние между которыми не превышает заданный радиус взаимодействия.

Полный перебор всех пар даёт простой и точный эталон, но имеет квадратичную стоимость. Пространственные индексы уменьшают число проверяемых кандидатов, однако их эффективность сильно зависит от текущей нагрузки.

Проект исследует компромисс между:

- **скоростью поиска** — задержка, пропускная способность и число проверенных кандидатов;
- **стоимостью обновления** — построение, переиспользование, refit и rebuild;
- **памятью** — хранение индекса и списков соседей;
- **устойчивостью** — поведение на равномерных, разреженных, кластеризованных и меняющихся сценариях.

## Исследовательский вопрос

> Может ли онлайн-политика выбирать пространственный индекс и стратегию обновления так, чтобы приближаться к производительности лучшего специализированного метода на изменяющихся нагрузках?

Позднее selector будет использовать дешёвые признаки: число частиц, разброс плотности, заполненность клеток, перемещение частиц, радиус поиска и результаты предыдущих измерений.

## Исследуемые подходы

| Семейство | Планируемые методы | Основная цель |
|---|---|---|
| Эталон корректности | полный перебор пар | получить точный список соседей |
| Регулярные сетки | плотная uniform grid | эффективно обрабатывать равномерные распределения |
| Разреженные сетки | hashed grid | не хранить пустые области пространства |
| Пространственный порядок | Morton/Z-order и непрерывные диапазоны клеток | улучшить локальность и уменьшить аллокации |
| Иерархические индексы | BVH | обрабатывать кластеры и неравномерные распределения |
| Динамические стратегии | reuse, refit, частичное обновление, rebuild | уменьшить стоимость каждого шага |
| Адаптивные политики | онлайн-выбор индекса и параметров | приближаться к лучшему методу для текущей нагрузки |
| Параллельные версии | multicore CPU и GPU | исследовать масштабирование и точки переключения между устройствами |

Ни один метод заранее не считается универсально лучшим. Цель — определить области нагрузок, где каждый подход конкурентоспособен, и сравнить адаптивную систему с фиксированными методами и oracle-baseline.

## Экспериментальные сценарии

Benchmark suite будет включать:

- равномерные распределения;
- разреженные пространства;
- один или несколько плотных кластеров;
- движущиеся кластеры;
- быстро меняющиеся распределения;
- разные числа частиц и радиусы поиска;
- позднее — variable-density и variable-radius сценарии.

Каждая оптимизированная реализация будет проверяться против полного перебора.

## Метрики

### Корректность

- точное совпадение списка соседних пар с brute force;
- поиск пропущенных и дублирующихся пар;
- проверка детерминизма там, где он требуется.

### Производительность

- время построения индекса;
- время reuse, update, refit и rebuild;
- задержка поиска соседей;
- число обработанных частиц и запросов в секунду;
- число проверенных кандидатов и найденных соседей;
- ускорение относительно brute force и фиксированных baseline.

### Память и масштабирование

- байты на частицу;
- память индекса и neighbor list;
- масштабирование по числу частиц;
- эффективность многопоточности;
- позднее — сравнение CPU и GPU.

### Качество адаптивного выбора

- точность выбора метода;
- стоимость сбора признаков и принятия решения;
- regret относительно oracle, заранее знающего самый быстрый метод;
- стабильность при изменении нагрузки.

## Методология

Эксперименты будут раздельно учитывать:

- генерацию workload;
- построение или обновление индекса;
- обход кандидатов;
- точные проверки расстояния;
- материализацию результата;
- проверку корректности;
- экспорт CSV и анализ.

Для каждого результата будут фиксироваться компилятор, флаги оптимизации, модель CPU/GPU, число потоков, количество частиц, размер пространства, радиус, параметры распределения, модель движения, прогрев, число повторений и статистика измерений.

## Текущий статус

**Проектирование исследования.**

Первая фаза будет CPU-only и намеренно узкой: 2D-частицы, полный перебор, uniform grid, тесты корректности и воспроизводимые benchmarks.

BVH, adaptive policies, GPU-ускорение и сложные стратегии обновления будут добавлены только после появления надёжной baseline-методологии.
